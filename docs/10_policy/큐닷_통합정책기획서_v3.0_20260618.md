# §3 010Q · OKQ 메일 시스템 (v3.0 개정)

> **핵심 명제:** 번호가 곧 이메일, 이메일이 곧 번호 — 무료(010Q)와 유료(OKQ)가 명확히 분리된다.

## §3-A 010Q메일 (무료 · 기본 등급)

| 항목 | 내용 |
|---|---|
| **가입 경로** | 홈페이지 접속 (추천링크 또는 QR코드) |
| **가입 조건** | 휴대폰 본인인증 |
| **발급 성격** | **인증 즉시 발급** — 선점 예약이 아님, 실사용 주소 즉시 활성 |
| **가격** | 무료 · 영구 |
| **이메일 형식** | `01070008000@Q.co.kr` (010 포함 11자리) |
| **타깃** | 전 국민 (휴대폰 보유자) |
| **Q-vic 적립** | ✅ 활동(출석·설문·추천 등)으로 획득 가능 |
| **Q-vic 사용** | ❌ 이니셜 구독(OKQ 플랜) 가입 후 사용 가능 |

### 가입 흐름

```
추천링크·QR 접속
    ↓
홈페이지 신청서 작성 (번호 입력)
    ↓
휴대폰 본인인증 (카카오 OAuth)
    ↓
010Q 메일 + Q-ID 즉시 발급 ← 선점·예약 없음, 실사용 주소 바로 활성
```

## §3-B OKQ메일 (유료 · 이니셜 구독 등급)

### OKQ 네이밍 의도

> **OKQ = "010, OK, Q"**
> - **010 붙여도·빼도 수신 OK** (7000-8000@Q.co.kr 과 010-7000-8000@Q.co.kr 모두 수신)
> - 명함에 번호 없이 이름만 쓰는 시대 — 이름@Q.co.kr 로 수신
> - **명함 사용자(직장인·전문가)** 특화 서비스

| 항목 | 내용 |
|---|---|
| **가입 경로** | 명함 스캔 (기본) 또는 010Q에서 구독 업그레이드 |
| **가격** | **월 ₩1,500 (= $1) · 이니셜 구독** |
| **이메일 형식** | `70008000@Q.co.kr` (010 제외 8자리) |
| **발급 원칙** | 8자리(010 제외)로 발급. 010Q(11자리)와 시각적으로 구분 → OKQ 등급 명함에서 즉시 식별 |
| **수신 호환** | `01070008000`, `010-7000-8000`, `70008000`, `7000-8000`, `010.7000.8000` 등 모든 표기 수신 |
| **구독 만료 시** | 8자리 주소(`70008000@Q`) 수신 차단 / 010Q 주소(`01070008000@Q`)는 무료로 계속 유효 |
| **내이름메일** | `김한국@Q.co.kr` — 한글 이름 기반 자동 예약 (양도 가능) |
| **영문여권이름메일** | `KIMHANKOOK@Q.co.kr` — 여권 기준 대문자 추가 생성 (양도 가능) |
| **타깃** | 직장인·전문가·명함 보유자 |
| **Q-vic 사용** | ✅ 취득 및 자유 사용 |
| **Q-bit 사용** | ✅ 취득 및 Q-bit 전환 |

# §13 Q-ID 통합 로그인 및 SSO 정책 (v3.0 신설)

## §13-A 통합 Q-ID 개념 및 방향성
1. **비밀번호 없는 로그인(Passwordless)**: 사용자가 별도의 비밀번호를 생성하거나 관리하지 않고, 이미 신뢰할 수 있는 소셜 계정(Google, Naver 등)과 휴대폰 번호(V-ID-010)를 매핑하여 안전하고 즉각적으로 본인을 증명한다.
2. **멀티 소셜 계정 연동**: 한 명의 Q-ID 사용자는 Google, Naver, Kakao 계정을 각각 중복하여 연동하고 해제할 수 있다.
3. **계정의 주체**: 계정의 주 식별자는 `phone`(휴대폰 번호)이며, 이에 대응되는 `q_id`가 발급된다. 소셜 계정은 Q-ID라는 본인 계정에 접근하기 위한 부차적인 '열쇠(Credential)'로 취급된다.

## §13-B 연동 및 세션 매핑 규칙

### 1. 자동 매핑 (1-Tap 연동)
*   **원칙**: 사용자가 Google 또는 Naver로 로그인을 시도할 때, 해당 소셜 계정의 이메일이 Q-ID 가입 신청서에 등록한 명함 이메일(`connect_email`) 또는 보조 이메일(`kor_email`, `eng_email`)과 완벽히 일치하고 상태가 `active`인 경우, 시스템은 별도의 본인확인 과정 없이 해당 계정의 소유자임을 간주하고 **자동으로 Q-ID에 연동(UID 업데이트)** 처리한다.
*   **예외**: 이미 해당 소셜 UID가 다른 Q-ID에 연동되어 있는 경우에는 자동 매핑을 거부하고 오류를 반환한다.

### 2. 수동 연동 (카카오 2차 검증)
*   **원칙**: 소셜 로그인에 성공했으나 해당 소셜 이메일이 DB상의 어떤 Q-ID 이메일과도 매치되지 않는 경우, 프론트엔드는 "Q-ID 계정 연동" 화면을 띄운다.
*   **흐름**:
    1. 사용자가 본인의 Q-ID 휴대폰 번호(V-ID-010)를 입력한다.
    2. 입력한 번호가 DB에 존재하는 유효한 계정인지 1차 확인한다.
    3. 기존에 구축된 카카오 OAuth 인증 흐름의 `state` 파라미터에 compound 토큰(`[휴대폰번호]|[소셜프로바이더]|[소셜UID]`)을 탑재하여 인증 요청을 보낸다.
    4. 카카오 인증 성공 시 리다이렉트 콜백(`/auth/kakao/callback`)에서 해당 phone 레코드의 소셜 연동 컬럼(`google_uid` 또는 `naver_uid`)에 소셜 고유 ID를 안전하게 기록하고 연동을 완료한다.

## §13-C 서브도메인 간 싱글 사인온 (SSO) 정책
1. **루트 도메인 공유**: `q.co.kr`, `010.q.co.kr`, `today.q.co.kr` 등 모든 패밀리 사이트가 `*.q.co.kr` 루트 도메인을 공유한다.
2. **공유 쿠키 세션**: Supabase Auth Client 세션 쿠키의 `domain` 설정을 `.q.co.kr`로 고정한다.
3. **SSO 사용자 경험**: 사용자가 어느 한 서브도메인 서비스에서 로그인에 성공하면, 쿠키 세션이 브라우저 수준에서 공유되므로 다른 서브도메인 서비스에 처음 접근할 때 추가 로그인 과정 없이 상단 네비게이션 프로필 및 개인화 데이터가 자동으로 동기화된다.
4. **글로벌 로그아웃**: 어느 서브도메인에서든 로그아웃을 실행하면 `.q.co.kr` 도메인에 저장된 세션 쿠키가 파기되어 모든 패밀리 서비스에서 일제히 로그아웃 상태로 안전하게 전환된다.
