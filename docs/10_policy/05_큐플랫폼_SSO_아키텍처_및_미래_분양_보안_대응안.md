# 큐플랫폼 SSO 아키텍처 및 미래 분양 시 보안 대응 방안 (v1.0)

본 문서는 `q.co.kr`을 중심으로 하는 큐플랫폼 통합 로그인(Single Sign-On, SSO)의 설계 방식과, 향후 서브도메인을 외부 수분양자에게 분양하여 독자적인 사이트로 개방할 때 발생할 수 있는 보안적·기술적 한계 및 그에 대한 미래 준비 대응안을 기록합니다.

---

## §1. 현재 단계: 쿠키 공유 기반 폐쇄형 SSO 설계 (q.co.kr 중심)

현재 단계에서는 모든 패밀리 도메인(`q.co.kr`, `010.q.co.kr`, `today.q.co.kr` 등)의 소스코드를 큐플랫폼 본사에서 100% 통제하므로, **가장 구현이 빠르고 인프라 비용이 적은 쿠키 공유 방식**을 채택합니다.

### 1) 동작 메커니즘
1. **통합 로그인 일원화**: 모든 서브도메인에서 [로그인] 요청 시, 루트 도메인(`q.co.kr/login`)으로 리다이렉트하여 로그인 처리를 일임합니다.
2. **공유 쿠키 발행**: Supabase Auth 세션 정보를 브라우저 쿠키 스토리지에 기록하되, 쿠키의 도메인을 **`.q.co.kr`** (앞에 마침표 포함)로 명시하여 모든 2차 서브도메인 간 공유되도록 설정합니다.
3. **서브도메인 자동 로그인**: 사용자가 다른 서브도메인에 접속하면 `auth.js` 모듈이 공유 쿠키를 즉각 스캔하여 세션을 복원하고, 백엔드 `/api/auth/session-sync`와 동기화하여 UI를 활성화합니다.

## §2. 미래 단계: 서브도메인 분양 시 보안 위협 및 아키텍처 한계

향후 수분양자가 독자적인 코드 및 어플리케이션을 서브도메인(`user.q.co.kr` 등)에 탑재하여 운영할 경우, 단순 쿠키 공유 방식은 심각한 보안 구멍을 유발합니다.

### 1) 세션 하이재킹 및 XSS (Session Hijacking via XSS)
*   **보안 취약점:** `.q.co.kr` 도메인 아래 구워진 로그인 세션 쿠키는 스크립트(`document.cookie`)로 접근 가능합니다. 만약 특정 수분양자가 본인의 분양 사이트에 악의적인 XSS(Cross-Site Scripting) 코드를 심어놓으면, 해당 사이트에 접속한 다른 정상 회원의 로그인 토큰을 손쉽게 갈취할 수 있습니다.
*   **대비책:** 
    *   Supabase 세션 쿠키를 스크립트로 읽을 수 없도록 **`HttpOnly` 및 `Secure`** 옵션을 필수로 적용합니다.
    *   `HttpOnly` 적용 시 프론트엔드 SDK가 세션을 직접 핸들링하지 못하므로, 각 서브도메인의 백엔드 프록시(Cloudflare Workers/Pages Functions)가 대신 세션 쿠키를 읽어 Supabase API를 대리 호출하는 미들웨어 구조로 전환해야 합니다.

### 2) 개인정보 무단 수집 및 RLS 우회 리스크
*   **보안 취약점:** 자동 로그인이 편리하지만, 수분양자 사이트의 스크립트가 세션이 맺어진 사용자의 민감한 정보(이름, 이메일, 전화번호 등)를 임의의 백엔드로 자동 수집해 갈 우려가 있습니다.
*   **대비책:**
    *   **OAuth 2.0 / OpenID Connect (OIDC) 도입:** 카카오나 구글 로그인처럼, 서브도메인이 메인 `q.co.kr`에 명시적으로 로그인 권한 및 정보 제공 동의를 요청하고, 사용자가 동의(Consent) 버튼을 클릭했을 때만 한정된 사용자 프로필을 반환하는 표준 권한 부여 프로토콜로 업그레이드해야 합니다.
    *   Supabase의 Row Level Security(RLS) 정책을 통해 수분양자용 토큰의 데이터 읽기 권한을 해당 서브도메인 범위 내로 철저히 격리합니다.

### 3) 커스텀 도메인(CNAME) 적용 시 쿠키 단절
*   **기술적 한계:** 수분양자가 본인의 서브도메인에 고유 브랜드 도메인(예: `my-brand.com`)을 씌울 경우, 브라우저 Same-Origin 정책에 의해 `.q.co.kr` 쿠키는 전송되지 않아 자동 로그인이 즉시 중단됩니다.
*   **대비책:**
    *   커스텀 도메인 환경에서는 쿠키를 직접 공유할 수 없으므로, OIDC의 리다이렉션 모델을 통해 인증 토큰(Auth Code)을 전달받아 자체 도메인 세션을 성립시키는 프로토콜을 필수 구현해야 합니다.

## §3. 중장기 SSO 아키텍처 로드맵

| 단계 | 운영 방식 | SSO 구현 기술 | 주요 보안 조치 |
| :--- | :--- | :--- | :--- |
| **Phase 1 (현재)** | 폐쇄형 (본사 통제) | `.q.co.kr` 와일드카드 쿠키 공유 | 스토리지 청소 및 비인가 룩업 API 차단 |
| **Phase 2 (준비)** | 세미 개방형 (수분양자 유입) | `HttpOnly` 쿠키 복원 + 프록시 검증 | 스크립트 쿠키 탈취 원천 차단 |
| **Phase 3 (완전 개방)** | 개방형 (커스텀 도메인 허용) | OAuth 2.0 / OIDC 기반 인증 허브화 | 사용자 정보 제공 동의(Consent Screen) 의무화 |
