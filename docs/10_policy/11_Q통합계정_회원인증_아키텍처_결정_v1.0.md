# Q. 통합계정 회원·인증 아키텍처 결정 v1.0

> 확정일: 2026-07-17  
> 상태: 확정(향후 모든 Q.co.kr 서비스 개발에 적용)  
> 시스템 본체: `services/010/`

## 1. 확정 결정

010Q를 Q.co.kr 전체의 단일 회원관리 및 로그인 시스템으로 유지한다.

- 010Q는 Q 통합회원 원장, Q-ID, 인증수단, 세션, 계정 상태와 서비스 이용권한을 관리한다.
- Q.co.kr 아래의 모든 현재·신규 서비스는 별도 회원원장과 별도 로그인 체계를 만들지 않고 010Q 통합인증을 사용한다.
- 내부 시스템과 코드의 소유 경계는 010Q이지만, 사용자에게 노출하는 공식 명칭은 **`Q. 통합계정`**, 로그인 화면 제목은 **`Q. 통합계정 로그인`**으로 통일한다.
- **`010Q로 로그인`** 또는 **`010.Q 로그인`**은 통합 로그인 명칭으로 사용하지 않는다. `010Q`는 무료 번호메일 플랜·서비스 명칭으로만 사용한다.

## 2. 책임 경계

### 010Q 통합인증 본체가 관리하는 것

- 불변 내부 회원 식별자 `account_id`
- Q-ID 및 대표 휴대전화번호
- 카카오 등 외부 인증수단 연결
- 패스키 공개키와 기기 목록
- Q 통합인증 세션과 세션 폐기
- 회원 상태(활성·정지·탈퇴·복구 중)
- 관리자 역할과 감사로그
- 서비스별 가입·이용권한

### 개별 Q 서비스가 관리하는 것

- 해당 서비스의 도메인 데이터와 설정
- 010Q 통합인증에서 받은 `account_id`에 귀속되는 서비스 프로필
- 해당 서비스 전용 세션

개별 서비스는 회원 비밀번호, 카카오 사용자 연결, 패스키를 별도로 저장하거나 회원을 중복 생성하지 않는다.

## 3. 사용자 명칭

| 구분 | 공식 표기 |
|---|---|
| 로그인 화면 제목 | `Q. 통합계정 로그인` |
| 로그인 안내 | `Q. 통합계정으로 모든 Q 서비스를 이용합니다.` |
| 회원 설정 | `Q. 통합계정 관리` |
| 로그아웃 | `Q. 통합계정 로그아웃` 또는 문맥상 `로그아웃` |
| 무료 번호메일 상품 | `010Q` 또는 `010Q메일` |

## 4. 베타기간 인증 정책

베타기간의 비밀번호 없는 기본 흐름은 다음으로 확정한다.

```text
최초 가입·계정 연결
→ 카카오 OAuth code 서버 검증
→ 카카오 고유 사용자 ID를 해당 가입 트랜잭션에 연결
→ Q. 통합계정 생성 또는 기존 계정 연결
→ 패스키 등록 제안

평상시 로그인
→ 패스키(지문·얼굴·화면 잠금)

패스키 미등록·새 기기
→ 카카오로 계속하기

기존회원 예외
→ 기존 비밀번호를 숨겨진 전환기 비상수단으로 한시 유지
```

신규회원에게 전화번호 앞 4자리 초기 비밀번호를 자동으로 열지 않는다. 기존 초기 비밀번호는 명시적으로 허용된 레거시 회원에게만 제한적으로 적용한다.

010Q 번호메일은 희소 번호의 선점 상품이 아니므로, 카카오 전화번호와 가입서의 대표번호 일치는 베타 가입 차단 조건으로 사용하지 않는다. 카카오 제공 번호는 가능한 경우 참고 스냅샷으로만 저장하고, 계정 로그인은 검증된 카카오 고유 사용자 ID와 패스키를 기준으로 한다. 번호 소유 확인 상태는 별도 필드로 관리하며 2026년 10월 이후 SMS/PASS 도입 시 승격할 수 있게 한다.

## 5. 2026년 10월 이후 인증 로드맵

- SMS 또는 PASS를 최초 번호 확인, 번호 불일치, 새 기기와 계정 복구 수단으로 추가한다.
- 평상시 로그인 기본은 계속 패스키로 유지한다.
- 카카오는 편리한 가입·복구 선택지로 유지하되 필수 종속성에서는 단계적으로 제외할 수 있게 한다.
- 초기 4자리 비밀번호를 완전히 폐지한다.

## 6. Q.co.kr 전체 SSO 원칙

각 Q 서비스는 로그인이 필요할 때 010Q 통합인증 본체로 사용자를 보낸다.

```text
Q 서비스
→ Q. 통합계정 로그인(010Q 인증 본체)
→ 패스키 또는 카카오 인증
→ 1분 이내·1회용 인증코드 발급
→ 원래 Q 서비스의 허용된 callback으로 복귀
→ 서비스가 코드를 교환하고 서비스 전용 세션 발급
```

- 통합인증의 정식 진입 주소는 구현 시 `q.co.kr/login`, `account.q.co.kr` 또는 `010.q.co.kr/login` 중 하나로 확정하되, 모두 동일한 010Q 인증 본체로 수렴시킨다.
- 사용자 화면의 브랜드는 주소와 무관하게 `Q. 통합계정 로그인`이다.
- 허용 목록에 등록된 callback 주소로만 복귀한다.
- URL에 회원정보·전화번호·세션토큰을 직접 전달하지 않는다.
- `.q.co.kr` 공용 인증 쿠키 하나를 모든 서비스에 전달하는 방식은 사용하지 않는다.
- 010Q 통합인증 세션과 각 서비스 세션을 분리하고 통합 로그아웃·강제 세션 폐기를 지원한다.

## 7. 관리자 계정

- 관리자 역할은 `010q`·`okq`·`founder` 플랜과 분리한다.
- 대표 관리자와 운영 관리자는 패스키를 필수로 사용한다.
- 대표 관리자는 주 기기 패스키와 백업 패스키 또는 보안키를 각각 등록한다.
- 관리자 중요 작업은 재인증하고 모든 변경을 감사로그에 남긴다.
- 관리자 인증정보를 시연 대상자의 휴대전화에 입력하지 않는다.

## 8. 개발 적용 체크리스트

새 Q 서비스 또는 인증 관련 기능을 개발할 때 다음을 확인한다.

- [ ] 별도 회원원장을 만들지 않고 Q. 통합계정의 `account_id`를 사용했는가?
- [ ] 화면에 `Q. 통합계정 로그인`으로 표시했는가?
- [ ] `010Q로 로그인` 또는 `010.Q 로그인`이라는 사용자 문구를 만들지 않았는가?
- [ ] 서비스 자체 비밀번호·카카오·패스키 저장소를 만들지 않았는가?
- [ ] 등록된 callback과 일회용 코드 교환으로 서비스 세션을 발급하는가?
- [ ] 관리자 역할과 상품 플랜을 분리했는가?
- [ ] 로그인·연결·복구·관리자 작업이 감사 가능하게 기록되는가?

## 9. 기존 구현 전환 원칙

현재 `services/010/`의 Q-ID/전화번호+비밀번호 및 HMAC 세션 구현은 전환 기반으로 보존한다. 이후 개발 순서는 다음을 기준으로 한다.

1. 일반 가입용 카카오 콜백에 OAuth code와 일회용 state 실검증 적용
2. 카카오 고유 사용자 ID를 Q. 통합계정에 연결
3. 신규회원 초기 4자리 비밀번호 자동 허용 중단
4. 패스키 등록·로그인 추가
5. 불변 `account_id`와 통합계정 원장 분리
6. 서비스 등록, 일회용 인증코드, callback 교환 구현
7. 각 Q 서비스를 Q. 통합계정 로그인에 순차 연결
8. SMS/PASS 복구 도입 후 레거시 초기 비밀번호 폐지
