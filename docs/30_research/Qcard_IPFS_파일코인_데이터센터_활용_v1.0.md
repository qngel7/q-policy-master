## 2. GPU 비교: RTX 5060 Ti 16GB × 2 vs Tesla T4 16GB × 2

| 항목 | RTX 5060 Ti × 2 | Tesla T4 × 2 |
|---|---|---|
| 아키텍처 | Blackwell (2025) | Turing (2018) |
| 총 VRAM | 32GB GDDR7 | 32GB GDDR6 |
| 메모리 대역폭 | 896 GB/s | 640 GB/s |
| 소비전력 (2장) | ~360W | ~140W |
| 쿨링 | 액티브 (팬) | 패시브 (팬 없음) |
| LLM 추론 속도 | ~50-60 t/s (2장) | ~25-35 t/s (2장) |
| 최대 모델 (Q4) | ~70B 가능 | ~30-40B 권장 |
| 장당 가격 | $429 MSRP (신품) | $100~300 (중고) |
| 설치 환경 | 일반 ATX PC | 서버 섀시 필수 |

### 결론: 환경별 분리 운용

- **사무실 서버 섀시 (데이터센터)** → Tesla T4 × 2 (저전력 24/7, n8n 자동화 백엔드)
- **집 ATX PC** → RTX 5060 Ti × 2 (빠른 추론, 개발·테스트)

## 3. 파일코인 데이터센터 → Q. Platform 활용 방안

### 3-1. 파일코인 마이닝 시스템 구성

| 구성요소 | 역할 | 특성 |
|---|---|---|
| AMD EPYC CPU | PreCommit1 (PC1) | SHA 확장 필수 |
| GPU | PreCommit2 / Commit2 (PC2/C2) | 유휴 시간 존재 |
| NVMe SSD | 스크래치 공간 (섹터당 ~450GB) | 병렬 처리용 |
| HDD (PB급) | 실링된 섹터 저장 | 직접 파일시스템 불가 |
| WindowPoSt | 24시간마다 섹터 증명 | GPU 점유, 데드라인 엄수 |

### 3-2. 활용 가능한 자원

| 자원 | Q. 활용처 | 주의사항 |
|---|---|---|
| GPU 유휴 시간 | Ollama LLM 추론, 임베딩 | PoSt 데드라인 충돌 방지 필수 |
| 고성능 CPU | n8n 자동화, 문서 임베딩 | 세일링 중 부하 고려 |
| HDD 여유 공간 | 원본 사진 IPFS 핀 저장 | 실링 섹터와 별도 마운트 |
| 전력/냉각 인프라 | 추가 서버 저비용 확장 | 상면 비용 ≈ 0 |
| 네트워크 | Q. 서비스 호스팅 | - |

### 3-3. GPU 우선순위 설정 (필수)

WindowPoSt 데드라인 충돌 방지를 위해 AI용 GPU와 세일링용 GPU를 물리적으로 분리하거나 `CUDA_VISIBLE_DEVICES`로 격리해야 한다. FIL 담보 손실 방지가 최우선.

## 4. Q.카드 스토리지 아키텍처 설계

### 4-1. Q.카드 구성 요소

```
Q.카드 = 원본 사진 (배경)
       + 날짜 텍스트 오버레이
       + 메모 텍스트
       + 필터 / Q. 브랜딩
       ─────────────────────
       → 렌더링 → 카드 출력물 (WebP, 단일 파일)
```

메모 텍스트는 Supabase DB에 별도 보관 (수백 바이트, 비용 무시).

### 4-2. 마켓 판매 구조

서비스의 핵심 축:
1. **공유**: 카드 출력물로 커뮤니티에 공유 (쇼케이스)
2. **판매**: 소유자가 원할 경우 원본 고화질 사진을 마켓에서 유료 판매

→ **원본 사진은 삭제 불가.** 마켓의 "상품" 그 자체.

### 4-3. 최종 스토리지 구조

```
사용자 업로드
     │
     ├─→ 렌더링 Worker → 카드 출력물 (WebP 1~2MB)
     │                        └─→ Cloudflare R2  [핫, 공유/미리보기]
     │
     └─→ 원본 사진 (WebP lossless, ~5MB)
              └─→ 자체 IPFS 핫 노드 핀  [데이터센터 HDD]
                       └─→ CID → Supabase DB 저장
```

### 4-4. 파일별 저장 전략

| 파일 | 저장소 | 보관 기간 | 월 비용 |
|---|---|---|---|
| 카드 출력물 (1~2MB) | Cloudflare R2 | 영구 | $0.015/GB |
| 원본 사진 (5~10MB) | 자체 IPFS 핫 노드 | 영구 | ≈ HDD 원가만 |
| 메모 + 메타데이터 | Supabase DB | 영구 | 무시 수준 |

## 5. IPFS vs Filecoin 실링 — 핵심 구분

| | Filecoin 실링 섹터 | IPFS 핫 노드 핀 |
|---|---|---|
| 접근 방식 | 언실링 필요 | 즉시 |
| 응답 시간 | 수 시간 | 수십ms |
| 용도 | 장기 보관 증명 (채굴) | 실시간 서빙 |
| Q. 활용 | 건드리지 않음 | 원본 사진 보관·서빙 |

**결론**: 파일코인 데이터센터의 HDD 여유 공간에 **별도 IPFS 노드**를 운영해 원본 핀. Filecoin 실링 섹터와 완전 분리.

### IPFS CID의 소유권 증명 기능

IPFS에 업로드된 파일은 내용 기반 고유 주소(CID)를 가진다. 변조 불가, 최초 업로드 증명 가능 → NFT 없이도 디지털 소유권 증명 가능. Q. 명예경제 철학과 정합.

