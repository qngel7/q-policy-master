# 010Q 기반 Q. 통합계정 비밀번호 없는 로그인 구현계획

> 작성일: 2026-07-19  
> 상태: 구현 승인(2026-07-19)  
> 대상: `services/010/`  
> 사용자 표기: `Q. 통합계정 로그인`

## 1. 목표

현재 Q-ID/휴대전화번호+비밀번호 로그인을 다음 구조로 전환한다.

```text
신규가입
→ 카카오 OAuth 계정 실검증
→ Q. 통합계정 생성 및 카카오 고유 ID 연결
→ 로그인 세션 발급
→ 패스키 등록

평상시 로그인
→ 패스키(지문·얼굴·화면 잠금)

패스키 미등록·새 기기
→ 카카오로 계속하기

기존회원
→ 기존 로그인 세션 또는 비밀번호로 최초 1회 카카오 연결
→ 이후 패스키·카카오만 사용
```

카카오가 제공한 전화번호와 가입서 번호의 일치는 베타 가입 조건으로 사용하지 않는다. 번호메일은 입력 번호 기반으로 발급하며, 카카오 번호는 참고 스냅샷으로만 저장한다.

## 2. 현재 구현에서 해결할 문제

1. 일반 가입 콜백이 OAuth authorization code를 서버에서 검증하지 않는다.
2. OAuth `state`에 전화번호를 직접 넣고 이를 가입 레코드 식별값으로 사용한다.
3. 일반 가입에서는 카카오 고유 사용자 ID를 저장하지 않아 이후 카카오 로그인이 불가능하다.
4. 신규회원도 `password_hash IS NULL`이면 전화번호 앞 4자리가 자동 초기 비밀번호가 된다.
5. 로그인 화면이 Q-ID/전화번호+비밀번호를 기본으로 표시한다.
6. 패스키 등록·검증 API와 공개키 저장소가 없다.
7. 현재 HMAC 세션은 Q-ID와 전화번호 중심이라 향후 Q.co.kr 전체 통합계정용 불변 `account_id`가 없다.
8. 이름이 `테스트`라는 이유만으로 영구 테스트 계정 예외가 작동한다.

## 3. 확정 정책

- 010Q가 Q.co.kr 전체 회원원장과 인증 본체를 담당한다.
- 사용자 화면에는 `Q. 통합계정 로그인`으로 표시한다.
- 신규회원에게 비밀번호와 초기 4자리 비밀번호를 발급하지 않는다.
- 카카오 OAuth 계정 실검증은 필수지만 카카오 전화번호 일치는 필수가 아니다.
- 하나의 카카오 고유 사용자 ID는 하나의 Q. 통합계정에만 연결한다.
- 번호 소유 확인과 로그인 신원 연결은 별개로 관리한다.
- 베타 일상 로그인은 패스키, 대체 로그인은 카카오로 한다.
- 기존 비밀번호는 기존회원의 최초 연결과 긴급 롤백을 위해 한시 유지한다.
- 관리자 계정은 패스키를 필수로 하고 카카오 단독 로그인을 허용하지 않는다.

## 4. DB 변경안

기존 데이터에 영향을 주지 않는 추가형 마이그레이션으로 진행한다.

### registrations 추가 컬럼

```text
account_id                 uuid unique not null
kakao_user_id              text unique
kakao_phone_snapshot       text
kakao_connected_at         timestamptz
auth_method                text
legacy_password_allowed    boolean default false
phone_verification_status  text default 'unverified'
auth_version               integer default 1
last_login_at              timestamptz
```

- 기존 행에는 `account_id`를 백필한다.
- 기존회원만 `legacy_password_allowed=true`로 설정한다.
- 신규회원의 `auth_method`는 `kakao` 또는 패스키 등록 후 `kakao_passkey`로 저장한다.
- `phone_verification_status`는 베타에서 `unverified` 또는 `kakao_snapshot`으로 두고, 10월 이후 `sms_verified`·`pass_verified`로 승격한다.

### q_auth_transactions

카카오 시작과 콜백을 안전하게 연결하는 단기 트랜잭션 테이블이다.

```text
id
state_hash
purpose             signup | login | link | recovery
registration_id
account_id
return_to
expires_at
used_at
created_at
```

- `state`는 32바이트 이상의 난수로 생성한다.
- 원문 대신 해시를 저장한다.
- 유효기간은 10분 이내로 한다.
- 성공·실패와 관계없이 재사용을 막는다.
- `return_to`는 허용된 Q 도메인 목록만 받는다.

### q_passkeys

```text
id
account_id
credential_id       unique
public_key
counter
transports
device_name
created_at
last_used_at
revoked_at
```

회원 한 명이 휴대폰·PC·백업기기 등 여러 패스키를 등록할 수 있게 한다.

### q_webauthn_challenges

```text
id
challenge_hash
account_id
purpose             register | authenticate
expires_at
used_at
created_at
```

등록·로그인 challenge는 5분 이내, 1회용으로 사용한다.

## 5. API 변경안

### 카카오 인증

```text
POST /api/auth/kakao/start
GET  /auth/kakao/callback
POST /api/auth/kakao/link
```

#### 시작

1. 가입 목적과 서버의 pending registration ID를 받는다.
2. 서버가 일회용 `state`를 생성한다.
3. `q_auth_transactions`에 저장한다.
4. 클라이언트는 서버가 반환한 state로 카카오 authorize를 시작한다.

#### 콜백

1. state 존재·만료·미사용 여부를 확인한다.
2. 카카오 authorization code를 access token으로 교환한다.
3. 카카오 `/v2/user/me`에서 고유 사용자 ID를 조회한다.
4. 동일 `kakao_user_id`의 중복 연결을 차단한다.
5. 가입 트랜잭션이면 정확한 pending registration에 카카오 ID를 연결하고 활성화한다.
6. 로그인 트랜잭션이면 `kakao_user_id`로 계정을 조회한다.
7. Q. 통합계정 세션을 발급하고 state를 사용 완료 처리한다.

전화번호는 제공되는 경우 `kakao_phone_snapshot`에만 저장하고 가입 성공 여부에는 사용하지 않는다.

### 패스키

```text
POST /api/auth/passkey/register/options
POST /api/auth/passkey/register/verify
POST /api/auth/passkey/login/options
POST /api/auth/passkey/login/verify
GET  /api/auth/passkey/list
POST /api/auth/passkey/revoke
```

- 등록 API는 로그인된 Q. 통합계정만 호출할 수 있다.
- `userVerification='required'`를 사용한다.
- RP 표시 이름은 `Q.`로 한다.
- 인증 ceremony는 010Q 인증 본체에서 중앙 처리한다.
- WebAuthn 응답 파싱과 서명 검증을 자체 축약 구현하지 않고 검증된 서버 라이브러리를 번들링하는 방식을 우선 검토한다.
- origin과 RP ID를 서버 허용 목록으로 엄격히 검증한다.

### 기존 비밀번호 API

`POST /api/login`은 다음 조건일 때만 허용한다.

```text
legacy_password_allowed = true
```

- 신규회원은 항상 `false`다.
- 카카오와 패스키 연결 완료 후 사용자 선택 또는 전환 종료 시 `false`로 바꾼다.
- `password_hash IS NULL`만으로 초기 4자리 로그인을 허용하지 않는다.
- 로그인 화면의 `다른 방법` 아래에서만 노출한다.

## 6. 화면 변경안

### 기본 로그인 화면

```text
Q.
통합계정 로그인

[ 지문·얼굴·화면 잠금으로 로그인 ]
[ 카카오로 계속하기 ]

다른 방법
· 기존 Q-ID와 비밀번호
```

- 브라우저가 패스키를 지원하지 않거나 등록된 패스키가 없으면 카카오 버튼을 우선 안내한다.
- 이미 Q 통합 세션이 있으면 로그인 화면을 건너뛴다.

### 가입 완료 화면

```text
Q. 통합계정 가입 완료

다음부터 더 빠르게 로그인하세요.
[ 지문·얼굴 간편로그인 설정 ]
[ 나중에 ]
```

- 베타 초기에는 패스키 등록을 권장하되 가입 완료를 막는 강제 단계로 만들지 않는다.
- 등록을 건너뛰면 다음 카카오 로그인 후 다시 제안한다.

## 7. 기존회원 전환

기존 계정에 카카오 ID가 없으므로 전화번호만으로 자동 연결하지 않는다.

허용하는 최초 연결 방법:

1. 이미 로그인된 기존 세션에서 `카카오 연결` 실행
2. 기존 비밀번호로 한 번 로그인한 직후 카카오 연결
3. 소수 베타회원의 관리자 확인 연결(감사로그 필수)

연결 후 패스키를 등록하고 이후 기본 로그인에서 비밀번호를 요구하지 않는다.

## 8. 번호 중복과 베타 운영

번호메일은 선점 상품이 아니며 번호 일치가 가입 조건은 아니다. 다만 동일 번호가 이미 등록되어 DB의 unique 제약과 충돌할 경우 자동으로 기존 계정에 연결하지 않는다.

```text
동일 번호 없음
→ 신규가입 진행

동일 번호 있음 + 같은 카카오 ID
→ 기존 Q 계정 로그인

동일 번호 있음 + 다른 카카오 ID
→ 중복 계정 자동 생성·자동 연결 금지
→ 베타 운영 확인 대상으로 기록
```

10월 이후 실제 번호 인증이 도입되면 번호 확인 상태와 중복 정리 정책을 별도로 적용한다.

## 9. 세션 정책

- 일반회원 세션: 30일을 목표로 하되 베타 초기에는 기존 14일 유지 후 안정성 확인 뒤 연장한다.
- 쿠키: HttpOnly, Secure, SameSite=Lax 유지
- 세션 claims에 `account_id`, `q_id`, `auth_version`을 포함한다.
- DB의 `auth_version`이 달라지면 기존 세션을 무효화한다.
- 로그아웃·카카오 연결 해제·계정 정지 시 세션 폐기가 가능해야 한다.
- 관리자 세션은 일반회원보다 짧게 운영한다.

## 10. 테스트·시연 계정 정비

- `이름=테스트`만으로 영구 테스트 계정이 되는 예외를 제거한다.
- 영구 테스트 계정은 관리자 발급 토큰 또는 명시적인 테스트 환경에서만 사용한다.
- 테스트 계정은 카카오 ID 중복, 보상, 추천 통계와 실제 메일 라우팅에서 분리한다.

## 11. 구현 순서

### 단계 A — 카카오 비밀번호 없는 로그인

1. DB 추가형 마이그레이션 작성
2. 카카오 start/state 트랜잭션 API 구현
3. 일반 콜백의 code·state·카카오 사용자 ID 실검증 구현
4. 신규가입 시 카카오 ID 연결과 세션 발급
5. 기존회원 연결 흐름 구현
6. 로그인 화면을 `카카오로 계속하기` 중심으로 변경
7. 신규회원 초기 4자리 비밀번호 차단

완료 시점부터 패스키가 없어도 카카오만으로 비밀번호 없는 가입·로그인이 가능하다.

### 단계 B — 패스키 로그인

1. WebAuthn 라이브러리·Cloudflare 빌드 호환성 검증
2. passkey/challenge 테이블과 API 구현
3. 가입 완료 후 패스키 등록 UI 추가
4. 로그인 첫 버튼을 패스키로 변경
5. 기기 목록·삭제 기능 추가
6. 관리자 계정 패스키 필수 적용

### 단계 C — Q.co.kr 전체 통합로그인 기반

1. 모든 세션의 기준을 `account_id`로 전환
2. 서비스 등록·허용 callback 목록 추가
3. 1분 이내 1회용 로그인 코드 발급·교환 구현
4. 신규 Q 서비스부터 Q. 통합계정 로그인 적용
5. 통합 로그아웃과 서비스별 세션 폐기 구현

## 12. 기능 플래그와 롤백

```text
AUTH_KAKAO_LOGIN_ENABLED
AUTH_PASSKEY_ENABLED
AUTH_LEGACY_PASSWORD_ENABLED
```

- DB 변경은 컬럼·테이블 추가형으로 수행해 롤백 시 기존 로그인으로 복귀할 수 있게 한다.
- 초기 배포는 내부 관리자와 지정 베타회원에게만 기능 플래그로 공개한다.
- 카카오 로그인 안정화 후 전체 베타회원에게 열고, 이후 패스키를 순차 공개한다.
- 레거시 비밀번호 데이터는 베타 종료 전까지 삭제하지 않는다.

## 13. 필수 검증 항목

- 변조·만료·재사용된 state 거부
- 위조·재사용 authorization code 거부
- 동일 카카오 ID의 복수 계정 연결 차단
- 카카오 번호 일치·불일치·미제공 모두 정상 가입
- 신규회원 초기 4자리 비밀번호 로그인 불가
- 기존회원 레거시 로그인과 카카오 연결 성공
- WebAuthn challenge 재사용·잘못된 origin·RP ID 거부
- 패스키 등록·로그인·복수기기·삭제 성공
- 카카오 취소·네트워크 실패 시 pending 가입 데이터 보존
- B 가입 → C 초대 → C 가입 추천관계 유지
- 기존 프로필·명함저장·친구초대 세션 기능 회귀 없음

## 14. 완료 기준

- 신규회원이 비밀번호 생성 없이 가입한다.
- 신규회원이 카카오 또는 패스키로 다시 로그인할 수 있다.
- 패스키 등록회원은 지문·얼굴·화면 잠금만으로 로그인한다.
- 카카오 번호와 가입번호가 달라도 가입이 차단되지 않는다.
- 신규회원에게 초기 4자리 비밀번호가 열리지 않는다.
- 기존회원은 계정을 잃지 않고 카카오·패스키로 전환할 수 있다.
- 사용자 화면 전체에 `Q. 통합계정 로그인`을 사용한다.
- 기존 가입·프로필·추천 흐름의 회귀 테스트가 통과한다.

## 15. 현장 시연 가입 원칙

- 관리자가 낯선 참가자의 전화번호를 받아 대신 입력하는 것을 기본 흐름으로 삼지 않는다.
- 관리자는 자신의 기기에서 패스키로 로그인하고 행사 공용 또는 1인용 가입 QR을 발급한다.
- 참가자는 자신의 휴대폰으로 QR을 스캔하고 개인정보 입력, 카카오 연결, 패스키 등록을 직접 완료한다.
- QR에는 전화번호를 넣지 않고 만료되는 불투명 `invite_token`만 넣는다.
- 명함 스캔 시연은 전화번호 기반 `join_phone` 대신 10분 이내·1회용 `claim_token`으로 참가자에게 인계하는 방향으로 전환한다.
- 행사 QR은 행사 ID, 추천 귀속 account_id, 발급 관리자, 만료시각, 사용제한을 기록한다.
- B 가입 후 C 초대 시연은 B의 로그인 세션에서 B 귀속 초대 QR을 새로 발급한다.
