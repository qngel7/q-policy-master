# Q. 통합계정 비밀번호 없는 로그인 변경 배포문

> 승인일: 2026-07-19  
> 승인 상태: 구현·순차 배포 승인  
> 대상 서비스: `services/010/`  
> 역할: Q.co.kr 전체 회원원장·통합인증 본체

> 구현 상태(2026-07-19): 카카오 OAuth 실검증, 신규가입 서버 증명,
> 카카오 로그인·기존계정 연결, 패스키 등록·로그인·목록·폐기,
> 레거시 비밀번호 전환 플래그, 관리자 전용 패스키·상호승인 복구까지 코드 반영 완료.

## 배포 목적

현재 Q-ID/휴대전화번호+비밀번호 중심 로그인을 비밀번호 없는 Q. 통합계정 로그인으로 전환한다.

```text
최초 가입·연결: 카카오 OAuth 계정 실검증
평상시 로그인: 패스키(지문·얼굴·화면 잠금)
새 기기·패스키 미등록: 카카오로 계속하기
기존회원: 기존 세션 또는 기존 비밀번호로 최초 1회 연결
```

010Q는 내부 회원·인증 시스템의 본체로 유지하되 사용자 화면에는 반드시 **`Q. 통합계정 로그인`**으로 표시한다. `010Q로 로그인`, `010.Q 로그인`이라는 표현은 사용하지 않는다.

## 변경 정책

1. 카카오 로그인은 전화번호 본인인증이 아니라 카카오 OAuth 계정과 고유 사용자 ID의 실검증으로 사용한다.
2. 카카오 제공 전화번호와 가입서 전화번호의 불일치는 가입 차단 조건이 아니다.
3. 번호메일은 입력 번호 기반으로 발급한다. 카카오 전화번호는 제공될 경우 참고 스냅샷으로만 저장한다.
4. 하나의 카카오 고유 사용자 ID는 하나의 Q. 통합계정에만 연결한다.
5. 신규회원에게 비밀번호 또는 전화번호 앞 4자리 초기 비밀번호를 발급하지 않는다.
6. 기존 비밀번호는 기존회원의 최초 연결과 롤백을 위한 전환기 수단으로만 유지한다.
7. 관리자는 패스키를 필수로 하며 카카오 단독 관리자 로그인을 허용하지 않는다.

## 1차 배포 — 카카오 기반 비밀번호 없는 로그인

### 서버

- `q_auth_transactions`와 통합계정 인증 컬럼의 추가형 DB 마이그레이션을 작성한다.
- `account_id`를 기존 회원에게 백필하고 신규 세션의 기준 식별자로 사용한다.
- `/api/auth/kakao/start`에서 32바이트 이상 난수 state를 만들고 원문 대신 해시를 저장한다.
- state에 전화번호를 직접 넣지 않는다.
- `/auth/kakao/callback`에서 다음을 모두 검증한다.
  - state 존재·만료·미사용
  - authorization code의 서버측 token 교환
  - 카카오 `/v2/user/me` 응답
  - 카카오 고유 사용자 ID 중복 연결 여부
- 가입 callback은 state에 연결된 정확한 pending registration만 활성화한다.
- 로그인 callback은 `kakao_user_id`로 기존 Q. 통합계정을 찾고 세션을 발급한다.
- 세션 claims에 `account_id`, `q_id`, `auth_version`을 포함한다.
- 신규회원은 `legacy_password_allowed=false`로 생성한다.
- 기존 `/api/login`은 `legacy_password_allowed=true`인 계정만 허용한다.
- 비밀번호를 모르는 기존회원은 로그인 화면에서 가입번호를 입력하고 카카오 등록 번호 일치 확인으로 계정을 복구한다.
- 복구 성공 시 같은 카카오 고유 ID를 해당 Q. 계정에 연결하고 즉시 통합계정 세션을 발급한다.
- `password_hash IS NULL`만으로 초기 4자리 로그인을 허용하지 않는다.
- `이름=테스트`만으로 영구 테스트 계정에 귀속되는 공개 예외를 제거한다.

### 화면

로그인 기본 화면을 다음 순서로 변경한다.

```text
Q.
통합계정 로그인

[ 카카오로 계속하기 ]

다른 방법
· 기존 Q-ID와 비밀번호
```

- 신규가입 완료 즉시 Q. 통합계정 세션을 발급한다.
- 카카오 번호 불일치나 번호 미제공을 오류로 표시하지 않는다.
- 동일 전화번호가 다른 카카오 ID로 이미 등록된 경우에는 자동 연결하지 않고 운영 확인 상태로 안내한다.

### 기존회원 연결

- 이미 로그인된 기존 세션에서 카카오 연결
- 또는 기존 비밀번호로 한 번 로그인한 직후 카카오 연결
- 소수 베타회원은 감사로그가 남는 관리자 확인 연결

전화번호만 입력하여 미연결 기존 계정에 카카오 ID를 자동 연결하지 않는다.

## 2차 배포 — 패스키 로그인

### 서버

- `q_passkeys`, `q_webauthn_challenges` 테이블을 추가한다.
- 다음 API를 구현한다.

```text
POST /api/auth/passkey/register/options
POST /api/auth/passkey/register/verify
POST /api/auth/passkey/login/options
POST /api/auth/passkey/login/verify
GET  /api/auth/passkey/list
POST /api/auth/passkey/revoke
```

- WebAuthn 응답과 서명은 Cloudflare Web Crypto 기반 서버 검증 모듈로 검증한다.
- `userVerification='required'`, 표시 이름 `Q.`, 엄격한 origin·RP ID 허용 목록을 사용한다.
- challenge는 5분 이내·1회용으로 처리한다.
- 회원당 복수 패스키 등록과 개별 폐기를 지원한다.

### 화면

가입 또는 카카오 로그인 완료 후 다음 화면을 제공한다.

```text
다음부터 더 빠르게 로그인하세요.
[ 지문·얼굴 간편로그인 설정 ]
[ 나중에 ]
```

패스키 공개 후 로그인 화면은 다음 순서로 전환한다.

```text
[ 지문·얼굴·화면 잠금으로 로그인 ]
[ 카카오로 계속하기 ]

다른 방법
· 기존 Q-ID와 비밀번호
```

## 현장 시연 흐름

- 관리자는 참가자의 전화번호를 받아 대신 가입하지 않는다.
- 관리자가 자신의 기기에서 행사 공용 또는 1인용 가입 QR을 발급한다.
- 참가자는 자신의 휴대폰에서 입력·카카오 연결·패스키 등록을 완료한다.
- 가입 URL에는 전화번호를 넣지 않고 만료되는 불투명 `invite_token`을 사용한다.
- 명함 스캔 인계는 `join_phone` 대신 10분 이내·1회용 `claim_token`을 사용한다.
- B가 가입 후 C를 초대할 때는 B 계정에서 B 귀속 초대 QR을 발급한다.

## 기능 플래그

```text
AUTH_KAKAO_LOGIN_ENABLED
AUTH_PASSKEY_ENABLED
AUTH_LEGACY_PASSWORD_ENABLED
```

현재 기능 플래그는 서비스 전체 단위다. 초기에는 짧은 점검 시간에만 ON하여 관리자·지정 베타회원으로 실기기 확인하고, 이상이 없을 때 전체 베타회원에게 유지한다. DB 변경은 추가형으로 수행하며 레거시 비밀번호 데이터는 베타 종료 전까지 삭제하지 않는다.

### Cloudflare Pages 환경변수

기존 값을 유지한다.

```text
SUPABASE_URL
SUPABASE_SERVICE_KEY
SESSION_SECRET
KAKAO_REST_API_KEY
KAKAO_CLIENT_SECRET  # 카카오 앱에서 Client Secret을 사용하는 경우
```

이번 배포에서 다음 값을 명시한다.

```text
AUTH_KAKAO_LOGIN_ENABLED=false     # 최초 배포 직후 점검 전
AUTH_PASSKEY_ENABLED=false         # 최초 배포 직후 점검 전
AUTH_LEGACY_PASSWORD_ENABLED=true  # 기존회원 롤백 경로 유지
PASSKEY_RP_ID=q.co.kr
PASSKEY_ORIGIN=https://010.q.co.kr
PASSKEY_ALLOWED_ORIGINS=https://010.q.co.kr
```

카카오 개발자 콘솔의 Redirect URI에는 다음 주소가 등록되어 있어야 한다.

```text
https://010.q.co.kr/auth/kakao/callback
```

## 필수 검증

- 위조·만료·재사용 state와 authorization code 거부
- 동일 카카오 ID의 복수 계정 연결 차단
- 카카오 번호 일치·불일치·미제공 가입 성공
- 신규회원 초기 4자리 비밀번호 로그인 실패
- 기존회원 계정 연결 및 재로그인 성공
- 기존회원 비밀번호 분실 → 카카오 번호 일치 복구 성공 및 번호 불일치 거부
- 패스키 등록·로그인·복수기기·폐기 성공
- WebAuthn challenge 재사용, 잘못된 origin·RP ID 거부
- B 가입 → C 초대 → C 가입 추천관계 유지
- 기존 프로필·명함저장·친구초대 회귀 없음
- 관리자 패스키와 일반회원 권한 분리
- 관리자 일반 비밀번호·카카오 로그인 차단, 상호복구 링크와 오프라인 복구코드 검증

## 배포 순서와 중단 조건

1. Supabase에서 중복 카카오 ID가 없는지 사전 조회한다.
2. `services/010/db/add_passwordless_auth.sql`을 SQL Editor에서 실행한다.
3. 위 Cloudflare 환경변수를 저장하되 두 간편로그인 플래그는 `false`로 둔다.
4. 서버·화면 코드를 `publish.bat`으로 배포한다.
5. `/api/session`, 기존 비밀번호 로그인, 기존 프로필·명함 경로를 점검한다.
6. `AUTH_KAKAO_LOGIN_ENABLED=true`로 바꾸고 신규가입·기존계정 연결·재로그인을 실기기에서 확인한다.
7. `AUTH_PASSKEY_ENABLED=true`로 바꾸고 Android 지문/화면잠금과 iPhone Face ID에서 등록·로그인·폐기를 확인한다.
8. 이상이 없으면 두 플래그를 `true`로 유지한다.
9. `Q통합계정_관리자패스키_초기등록_운영절차_20260719.md`에 따라 서대원·서경석의 정확한 `account_id`를 확인한다.
10. `q_set_admin_account`로 역할을 지정하고 서대원 최초 패스키 → 서경석 상호승인 등록 순서로 완료한다.
11. 두 관리자 모두 2개 이상의 관리자 패스키와 오프라인 복구코드를 준비한 뒤 운영을 시작한다.

사전 중복 확인 SQL:

```sql
SELECT kakao_uid, COUNT(*)
FROM registrations
WHERE kakao_uid IS NOT NULL
GROUP BY kakao_uid
HAVING COUNT(*) > 1;
```

자동 회귀 테스트:

```text
npm --prefix services/010 run test:auth
```

이 배포에서는 기존 `join_phone` 시연 흐름을 유지한다. 행사 공용
`invite_token`·1회용 `claim_token`은 별도 운영 기능으로 분리한다. 관리자 역할별
패스키 강제·상호복구는 `Q통합계정_관리자패스키_초기등록_운영절차_20260719.md`에 따라 함께 배포한다.

다음 중 하나가 발생하면 해당 기능 플래그를 즉시 OFF하고 기존 로그인으로 복귀한다.

- 기존 계정과 잘못된 카카오 ID 연결
- OAuth state 재사용 또는 callback 위조 가능
- 신규회원에게 초기 4자리 비밀번호가 열림
- 기존 가입·추천 데이터 손상
- 패스키 origin·challenge 검증 실패
